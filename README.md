### 08.07.2026 начал делать фрейминг. Формат сообщений TLV `[тип][длина][данные]`
Тип - 1 байт
Длина каждого сообщения - 2 байта, то есть до 65535 байт на один кадр.
Плюс network byte order, то есть big-endian (`b34f` -> `4fb3`).

### 19.07.2026 Закончил писать клиента на ESP32. Отправил тестовое сообщение по протоколу:
```c
        uint8_t payload = 0x42;     // Это тестовая нагрузка, пока не дописан протокол
        Header hdr = {
            .type = 0x01,
            .length = htons(sizeof(payload))
        };
```
Несмотря на попытку обработать паддинг на стороне сервера, все равно влез лишний байт при отправке.
Предположительно из-за разницы архитектур x86-64 и Xtensa.

### 20.07.2026 Добавил в структуру заголовка `__attribute__`, это исправило паддинг:
```c
typedef struct __attribute__((packed)){
    uint8_t type;
    uint16_t length;
} Header;
```
До этого фикса несмотря на размер payload длина сообщения всегда была 0.
На стороне книента добавил полезную нагрузку - MAC & IP. Каждое сообщение 13 байт:
- 3 байта заголовок
- 10 байт нагрузка
```c
typedef struct {
    uint8_t mac[6];
    uint8_t ip[4];
} ESP32_data;
```
Через несколько дней добавлю парсинг на сервере.
